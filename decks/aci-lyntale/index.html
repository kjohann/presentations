Azure Container Instances

  • Ingen VM'er
  • Ingen clusteroppsett
  • Bring your own container
  • Custom cpu
  • Custom minne
  • Betal pr sekund
  • Lar deg kjøre Linux (og Windows) containere uten VM'er eller andre høynivå tjenester som SF/Web Apps
  • Trenger ikke tenke på orkestrering/scheduling osv
  • Bare deploy et image - enkelt og greit
  • Kan styre cpu og minne (uavhengig - yay!)
  • Betaler kun for bruk
  • mao

Azure Container Instances er en tjeneste som lar deg kjøre containere, uten å tenke på VM'er, Web apps eller andre plattformer som Service Fabric. Alt du trenger er et image. Så kan du velge hvor mye cpu og minne du skal ha, uavhengig = digg. Det er støtte for både Linux og Windows, og du betaler kun for det du bruker.

Container group

  • az container create => Container Group
  • Deler VM og IP
  • Kan eksponere flere porter
  • Containere kan nå hverandre på localhost
  • Ingen portmapping pga delt IP
  • Kan sammenlignes med en pod i Kubernetes

Du får en container group. En group kan ha en public ip men kun en, men man kan tilgjengeliggjøre flere porter. Det er dog ikke støtte for portmapping. Lar deg provisjonere flere containere sammen i en gruppe som deler hostmaskin og nettverk. En container group er sammenlignbar med en pod i kubernetes. Betyr at du i teorien kan deploye en hel applikasjon bestående av flere mikrotjenester i en enkelt container group hvor du selv kan definere hvor mye ressurser hver enkelt container skal ha.

Orkestrering

 

  • Håndterer enkeltcontainere
  • Tradisjonelle orkestreringsløsninger på toppen
  • Kubernetes: https://github.com/Azure/aci-connector-k8s
  • I mikrotjenestearkitektur bruker man som regel orkestrering
  • Ingen egen skreddersydd orkestrator for ACI
  • ACI har minimale orkestreringsegenskaper for å håndtere enkeltcontainere
  • Skal kunne bruke eksisterende gjennom connectors
  • Betyr at man kan legge til ACI som en "uendelig" node i f.eks en kubernetes cluster
  • Vise hvordan det funker (håper jeg)

https://github.com/Azure/aci-connector-k8s I en større mikrotjenestearkitektur vil man som regel bruke en eller annen form for orkestrering som f.eks Kubernetes eller Docker Swarm. Istedenfor å lage en egen orkestrator for ACI har man lagt til rette for at eksisterende løsninger skal kunne utnytte elastiteten som ACI tilbyr gjennom såkalte connectors. Og istedenfor å forsøke å forklare hvordan det funker, tenkte jeg å prøve å vise.

Demo:

  • kubectl get nodes (vis cluster)
  • head -18 aci-connector.yaml (vil ikke vise secrets)
  • kubectl create -f aci-connector.yaml
  • kubectl get nodes -w
  • kubectl create -f nginx-pod.yaml
  • kubectl get po -w -o wide
